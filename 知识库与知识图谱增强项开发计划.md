# 知识库与知识图谱增强项开发计划

## 1. 背景

`知识库与知识图谱结构重构方案.md` 的阶段 1~8 已全部完成（Sprint 详情 Tabs 化为最后一项）。本计划覆盖该方案中标注为「后续增强」的 12 项能力，以及附录 17 中尚未落地的关键技术建议，按依赖关系和价值分 8 个批次推进。

每批次都是可独立交付、可回退的小闭环。排除项：附录 17.2 权限完善（暂不处理）。

## 2. 总体原则

- **复用现有架构**：FastAPI + SQLAlchemy + PostgreSQL + Vue 3 + Element Plus + Pinia，沿用现有 CRUD/API/Schema 模式与 RESTful `/api/v1` 规范。
- **TraceLink 为关系底座**：新增关系优先写 TraceLink，图谱与覆盖率从 TraceLink 派生，不重复造关系存储。
- **数据库迁移**：新增表/字段/索引走 Alembic 单链迁移，部署后**手动 `python -m alembic upgrade head`**（API 容器启动不自动跑）。
- **分批交付**：每批完成后做后端 import 烟雾测试 → 部署 → 服务器接口回归 → 前端验证，再进入下一批。
- **安全边界**：不在代码写公网 IP / 账号 / 密码；推送前做隐私检查；Push 不加 Claude 署名。

## 3. 增强项总览

| 批次 | 主题 | 涉及面 | 新表/字段 | 优先级 |
|------|------|--------|----------|--------|
| 批次 1 | 前端列增强（关联 API/脚本/功能点数） | 后端 Out 聚合 + 前端列 | 无 | 高（快赢） |
| 批次 2 | Sprint 聚合接口 + 统一覆盖率口径 | 后端新 API + 前端替换 N+1 | TraceLink 索引 | 高 |
| 批次 3 | 脚本/selector 精确映射 | 后端解析 service + 用例字段更新 | 无 | 高（自动化闭环） |
| 批次 4 | ImportJob 异步化 | 后端 Job 表 + worker + 前端轮询 | import_jobs 表 | 中 |
| 批次 5 | 图谱查询性能 + ECharts 交互 | 后端图谱 API + 前端图组件 | GIN 索引 | 中 |
| 批次 6 | 流水线智能化（AI） | 后端 prompt/service + 前端展示 | 无 | 中 |
| 批次 7 | sprint_all 脚本合并 + sequence_no | 后端 baseline + Sprint 字段 | sprints.sequence_no | 中 |
| 批次 8 | 数据库唯一约束补齐 | 数据清洗 + uq 约束迁移 | 约束 | 低（收尾） |

## 批次 1：前端列增强（关联 API / 脚本 / 功能点数）

### 目标
功能点矩阵补「关联 API 数」「脚本数」列；接口清单补「关联功能点」列。让矩阵真正反映"功能点↔接口↔脚本"三维覆盖。

### 后端
- `app/crud/crud_feature_point.py`：`get_feature_points` 的输出转换中，基于 TraceLink 统计：
  - `api_count`：`feature → api` 且 `relation_type in ('implements','covers')` 的去重 api 数。
  - `script_count`：`feature → script` 或经 testcase 间接关联的 script 数（`feature→testcase→script`）。
- `app/crud/crud_api_endpoint.py`：`get_api_endpoints` 输出补 `feature_count`：`api → feature` (`implements`) 去重数。
- 复用 `crud_trace_link` 的查询；新增 helper `count_related(db, entity_type, entity_id, target_type, relation_types)`。
- Schema：`FeaturePointOut` 加 `api_count`/`script_count`；`ApiEndpointOut` 加 `feature_count`。

### 前端
- `FeatureMatrix.vue`：表格加「关联 API」「脚本数」两列（复用现有 coverage_count 列样式）。
- `ApiEndpointList.vue`：表格加「关联功能点」列。

### 数据库
- 无新表/字段。仅依赖现有 `trace_links`。建议在本批次顺手补 `trace_links` 核心索引（见批次 2），避免 count 查询慢。

### 验证
- 功能点列表接口返回含 `api_count`/`script_count`；接口列表含 `feature_count`。
- 前端矩阵/清单正确展示数字。

### 风险
- TraceLink 数据稀疏时数字为 0（正常，需先跑流水线/导入产生关系）。

## 批次 2：Sprint 聚合接口 + 统一覆盖率口径

### 目标
消除前端 N+1 请求，统一全平台覆盖率口径（功能点 / API / 自动化 / 回归影响）。

### 后端

#### 聚合接口（新增 `app/api/v1/sprint_knowledge.py` 或并入 `sprints.py`）
- `GET /api/v1/sprints/{sprint_id}/knowledge-overview` → `{ asset_count, feature_count, testcase_count, api_count, script_count, feature_coverage, api_coverage, automation_coverage, change_count }`
- `GET /api/v1/sprints/{sprint_id}/feature-matrix` → 每个功能点的覆盖/关联API/关联脚本/状态。
- `GET /api/v1/sprints/{sprint_id}/api-coverage-matrix` → 每个 API 的关联功能点/关联用例/状态/优先级。
- `GET /api/v1/sprints/{sprint_id}/change-impact-matrix` → 每个变更影响的功能点/用例/API/脚本。

#### 统一覆盖率计算（新增 `app/services/coverage_calculator.py`）
- 功能点覆盖率 = 有 active `FeaturePointTestCase` 或 `feature→testcase covers` TraceLink 的功能点数 / active 功能点总数。
- API 覆盖率 = 有 `TestCaseApiEndpoint` 或 `testcase→api tests_api` 的 API 数 / active API 总数。
- 自动化覆盖率 = `testcases.automation_status in ('generated','validated','executed','healed')` 或有 `testcase→script` 的用例数 / 用例总数。
- 回归影响覆盖率 = `change_items.status in ('applied','ignored')` 或已关联新增/更新用例的变更数 / 变更总数。
- 复用现有 `get_coverage_count` / `get_testcase_count`，统一入口避免各页算法漂移。

### 前端
- `KnowledgeBase.vue` 首页统计卡改用 `knowledge-overview`（替换现有多接口拼接）。
- `CoverageAnalysis.vue` 改用聚合接口的覆盖率字段（保持现有 UI，数据源切换）。

### 数据库
补 TraceLink 关键索引（迁移）：
```
ix_trace_links_source_relation (source_type, source_id, relation_type)
ix_trace_links_target_relation (target_type, target_id, relation_type)
ix_trace_links_relation (relation_type)
```
（部分可能已存在，迁移中 `IF NOT EXISTS` 或先查信息表。）

### 验证
- 4 个聚合接口返回结构正确，覆盖率与逐条统计一致。
- 前端首页/覆盖分析加载请求数显著减少。

### 风险
- 大 Sprint 聚合查询慢 → 靠索引 + 限制 active 状态过滤缓解；必要时后续做物化视图。

## 批次 3：脚本 / selector 精确映射（自动化闭环）

### 目标
打通"用例→脚本"链路：导入 Playwright 脚本时解析 `test()` 标题里的 case_no，关联 TestCase，更新自动化状态。

### 后端

#### 新增解析服务 `app/services/script_parser.py`
- `parse_spec_file(path) -> list[{case_no, test_title, line}]`：正则 `([A-Za-z0-9]+_TC_[A-Za-z0-9]+_\d{3,})` 提 case_no。
- `parse_selector_imports(spec_path) -> selector_paths`：从 `import ... from './selectors/xxx'` 提 selector 关联。
- 无 case_no 时回退：标题与 `TestCase.title` 精确/相似度匹配（confidence=60）。

#### 集成点
- `local_project_importer.py`：导入 `test_script` 资产后，调用 script_parser：
  - 写 `testcase → script` TraceLink（`generated_by`），metadata 记 test_title/line。
  - 更新 `TestCase.automation_status='generated'`、`automation_path=spec_path`。
  - 写 `script → selector` TraceLink（`depends_on`）。
- `api/v1/knowledge_assets.py`：新增 `POST /knowledge-assets/{asset_id}/link-script` 手动触发单脚本解析（兜底）。

### 前端
- `LocalImport.vue`：导入结果展示「脚本关联用例数」。
- `TestCaseList.vue` 详情抽屉：展示 `automation_status` / `automation_path` / 关联脚本（TraceLink）。

### 数据库
- 无新表/字段。复用 `KnowledgeAsset(test_script/selector_map)` + `TraceLink` + `testcases.automation_status/automation_path`（阶段 1 已加）。

### 验证
- 导入含 case_no 的 spec.ts 后，对应 TestCase `automation_status=generated`、`automation_path` 有值，TraceLink 有 `testcase→script generated_by`。
- selector 关联 `script→selector depends_on`。

### 风险
- 历史 spec 无 case_no → 回退标题匹配，confidence 低，warnings 提示人工确认。

## 批次 4：ImportJob 异步化

### 目标
本地项目导入可能耗时（大量文件 + 解析），改为异步 Job + 进度可查，避免接口超时。

### 后端

#### 新增模型 `app/models/import_job.py`
```
ImportJob: id, project_id, root_path, job_type, status(pending/running/succeeded/failed/partial_success/cancelled),
dry_run, total_files, processed_files, success_count, failed_count,
result_summary(JSON), warnings(JSON), errors(JSON), created_at, started_at, completed_at
```

#### API（新增 `app/api/v1/import_jobs.py`）
- `POST /api/v1/import-jobs/local-project` → 创建 Job（status=pending），返回 job_id；后台触发执行。
- `GET /api/v1/import-jobs` / `GET /api/v1/import-jobs/{id}` → 查状态/进度/结果。
- `POST /api/v1/import-jobs/{id}/cancel` → 取消（running 时标记 cancelled）。

#### 执行器
- 用 FastAPI BackgroundTasks 或独立线程池跑 `LocalProjectImporter.import_project`，逐文件更新 `processed_files`。
- 单文件失败不中断，记 errors；结束按成功率置 succeeded/partial_success/failed。
- 保留现有同步 `POST /knowledge-assets/import-local-project` 作为 dry_run 快速预览（小目录），大目录走 Job。

### 前端
- `LocalImport.vue`：提交后拿 job_id，轮询 `GET /import-jobs/{id}`（5s 间隔），展示进度条（processed/total）+ warnings/errors 折叠。
- 完成后展示 result_summary（同现 dry_run 结果结构）。

### 数据库
- 新表 `import_jobs` + 索引（project_id, status, created_at）。迁移 `r9l2m3n4o5p6_add_import_jobs.py`。

### 验证
- 大目录导入不再超时；前端进度实时更新；中断/失败有明确状态。

### 风险
- 后台任务进程异常退出 → Job 卡 running；加 `started_at` 超时检测（如 30min 无更新置 failed）。

## 批次 5：图谱查询性能 + ECharts 交互

### 目标
图谱支持子图/邻居/搜索查询，前端引入真正的交互式图可视化。

### 后端

#### 图谱 API 扩展（`app/api/v1/graphs.py`）
- `GET /graphs/{id}/subgraph?node_types=&relation_types=&module_id=&center_type=&center_id=&depth=1&limit=300` → 子图（节点+边），超限返回 `truncated:true`。
- `GET /graphs/{id}/node/{node_id}/neighbors?direction=both&depth=1` → 一跳邻居。
- `GET /graphs/{id}/search?keyword=` → 节点搜索。
- 大模块（用例>100）返回聚合节点，点击再展开。

#### 性能
- `crud_graph` 查询基于 GraphNode/GraphEdge 表 + 索引（`graph_id+node_type`、`graph_id+relation_type`、`source_node_id`、`target_node_id`）。
- `trace_links` / `graph_nodes.properties` 补 GIN 索引（JSON 查询）。

### 前端

#### 引入图可视化库
- 选型：**ECharts graph**（与 Element Plus 生态兼容、bundle 适中）或 **AntV G6**（交互更强）。
- `graph/KnowledgeGraph.vue`：替换现列表+抽屉为力导向图，支持缩放/拖拽/点击高亮上下游/搜索定位。
- `KnowledgeDetail.vue` 图谱 Tab：复用图组件做 Sprint 子图嵌入（center=Sprint，depth=1）。

### 数据库
- 迁移补 GraphNode/GraphEdge 索引 + properties GIN。
- 迁移 `s10n3o4p5q6_add_graph_indexes.py`。

### 验证
- subgraph 接口按节点/关系类型过滤正确，超限 truncated。
- 前端图可交互，点击节点展示上下游，搜索定位。

### 风险
- 图库 bundle 体积 → 按需引入 echarts/graph 模块；超大图性能 → 强制 limit + 聚合。

## 批次 6：流水线智能化（AI）

### 目标
用 LLM 增强增量变更识别、按模块准备、接口-用例覆盖映射，减少规则误判。

### 后端

#### PRD 文本级 LLM 变更识别
- `app/services/prompts/skill_prompts.py`：新增 `build_change_detection_prompt(sprint_prd_text, baseline_features)`，要求 LLM 输出 `{added, modified, removed, clarified}` 带 evidence。
- `pipeline_executor._execute_stage_1`（incremental）：规则分析后，可选调用 LLM 复核/补充 ChangeItem（`source='ai_llm'`，confidence 标注）。
- 保留规则分析作为兜底（LLM 失败时）。

#### 按模块 prepare
- `sprint_baseline_manager.prepare_from_all` / `clone_baseline_modules_to_sprint` 支持 `module_ids` 参数，只克隆指定模块底稿。
- `api/v1/sprints.py` `prepare-incremental` 接口接收 `module_ids`（前端选择后传）。

#### 接口-用例智能覆盖
- 新增 `app/services/api_coverage_mapper.py`：对无 `tests_api` 关系的 testcase+api，调 LLM 判断是否覆盖，写 TraceLink（confidence 由 LLM 给，低置信需人工确认）。
- `api/v1/api_endpoints.py`：新增 `POST /api-endpoints/map-coverage`（按 sprint 批量映射）。

### 前端
- `AIWorkbench.vue` 增量模式：prepare 步骤展示识别的变更模块，用户勾选后再克隆。
- `ChangeItemList.vue`：变更项标注 `source`（规则/LLM）。
- `ApiEndpointList.vue`：「智能映射覆盖」按钮。

### 数据库
- 无新表。ChangeItem 复用 `source`/`confidence`；TraceLink 复用 confidence/evidence。

### 验证
- LLM 变更识别产出合理 ChangeItem；规则兜底正常。
- 按模块 prepare 只克隆勾选模块。
- 接口-用例映射产生 tests_api TraceLink。

### 风险
- LLM 误判 → 低置信关系前端标"待确认"，不直接污染基线；AI 配置缺失时降级为纯规则。

## 批次 7：sprint_all 脚本合并 + Sprint sequence_no

- 迁移 `t20p4q5r6s7_add_sprint_sequence_no.py`：加字段 + 按名称数字回填历史 Sprint。

## 批次 8：数据库唯一约束补齐

### 目标
数据清洗后补关键唯一约束，防止重复实体/关系，保障 upsert 幂等。

### 后端 / 数据库

#### 迁移前数据清洗（迁移内 `op.execute` 或独立脚本）
- `testcases`：同 `(project_id, sprint_id, case_no)` 去重，保留最新。
- `api_endpoints`：同 `(project_id, sprint_id, method, path)` 去重。
- `trace_links`：同五元组去重，合并 confidence/evidence。
- 回填 `testcases.sprint_id`（null 的按 case_no/asset 推断，无法确定保留 null）。

#### 迁移 `u30q5r6s7t8_add_unique_constraints.py`
```
uq_testcases_project_sprint_case_no (project_id, sprint_id, case_no)
uq_api_endpoints_project_sprint_method_path (project_id, sprint_id, method, path)
uq_trace_links_unique_relation (source_type, source_id, target_type, target_id, relation_type)
```
- 对含 null 的列不加严格 uq（PostgreSQL null 不参与唯一），或先回填再建。

### 验证
- 迁移成功，重复数据已清洗。
- 重复 upsert 不再产生重复行。

### 风险
- 清洗误删 → 迁移内先备份/打印将被清理的行；生产环境先在测试库验证。

## 4. 数据库迁移策略

1. 单链 Alembic，每批次一个迁移文件，`down_revision` 接当前 head。
2. 新字段 nullable + 默认值，不破坏历史数据。
3. 唯一约束（批次 8）必须先清洗再建，迁移内含清洗 SQL 并打印影响行数。
4. 索引（批次 2/5）用 `IF NOT EXISTS` 或先查信息表，幂等。
5. 部署后手动 `docker compose exec api bash -c "cd /code && python -m alembic upgrade head"`。

## 5. 风险与降级

| 风险 | 降级 |
|------|------|
| 聚合接口大 Sprint 慢 | 索引 + active 过滤；后续物化视图 |
| LLM 误判/不可用 | 规则兜底；低置信关系标"待确认" |
| 图库 bundle 体积 | 按需引入；超大图强制 limit + 聚合 |
| 异步 Job 卡死 | started_at 超时检测置 failed |
| 唯一约束迁移清错数据 | 迁移内打印影响行 + 测试库验证 |

## 6. 验证计划

- 后端：每个迁移链 import 烟雾测试（`from app.main import app`）、router 注册、Schema 序列化。

### 接口回归（部署服务器后）
- 批次 1：功能点/接口列表含新字段。
- 批次 2：4 个聚合接口返回正确，覆盖率一致。
- 批次 3：导入 spec 后 testcase.automation_status/TraceLink 更新。
- 批次 4：Job 创建/轮询/状态流转。
- 批次 5：subgraph/neighbors/search 接口。
- 批次 6：LLM 变更识别 + 按模块 prepare + 接口映射。
- 批次 7：脚本合并 + sequence_no 排序。
- 批次 8：约束建立 + 重复清洗。

### 前端手工
- 矩阵/清单新列、首页聚合统计、用例自动化状态、Job 进度、交互图谱、增量模块勾选。

## 7. 推荐实施顺序

1. **批次 1**（快赢，纯聚合，立即可见）。
2. **批次 2**（聚合接口 + 覆盖率统一，基础设施，后续批次受益）。
3. **批次 3**（脚本映射，打通自动化闭环，测试工程核心价值）。
4. **批次 7**（sequence_no + 脚本合并，补齐基线机制）。
5. **批次 6**（AI 智能化，提升增量/覆盖准确性）。
6. **批次 4**（异步导入，体验优化）。
7. **批次 5**（图谱交互，可视化提升，工作量最大）。
8. **批次 8**（约束收尾，数据治理）。

每批完成后更新 `知识库与知识图谱结构重构方案.md` 的进度说明，保持文档与代码一致。
