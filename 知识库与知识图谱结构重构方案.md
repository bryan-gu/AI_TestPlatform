# 知识库与知识图谱结构重构方案

## 1. 背景与目标

当前 TestFlow 已经具备 Sprint、Document、Module、FeaturePoint、TestCase、Graph 等基础能力，也能通过 AI 工作台完成需求分析、测试用例生成、脚本生成和执行自愈。但结合现有 SKILL 流水线和外部测试项目资料的真实结构来看，当前知识库仍偏向“文档管理 + AI 摘要”，还没有形成面向测试工程的完整知识资产网络。

本方案以 VAgent 作为样例项目进行分析。VAgent 不是后续平台的中心对象，只是用于验证 TestFlow 能否承接真实测试项目资料的参考样本。平台开发完成后，应以 TestFlow 内部的项目、Sprint、知识资产、测试用例和图谱为主，外部项目目录仅作为导入来源或迁移样本。

样例项目的核心知识资料主要集中在：

- `D:/BF-VAgent/需求文档`：PRD、Sprint 需求文档、会议纪要、UI 探索生成的功能点、`cases.json`。
- `D:/BF-VAgent/测试用例`：Excel 测试用例、按 sprint 组织的 Playwright 脚本、选择器、执行产物。
- `D:/BF-VAgent/接口文档`：Markdown API 规划、OpenAPI JSON，其中 `Admin API_OpenAPI.json` 包含大量 endpoint、schema、tag、operation 信息。

这些资料之间天然存在追踪关系：需求产生功能点，功能点生成测试用例，用例生成脚本，接口文档对应 API 节点，API 节点被用例覆盖，Sprint 变更影响已有功能点、用例、接口和脚本。当前结构没有把这些关系建成一等对象，导致知识图谱难以做真正的影响分析和覆盖分析。

本方案目标：

1. 将知识库从“文档列表”升级为“测试知识资产库”。
2. 将需求、功能点、测试用例、接口、脚本、执行结果纳入统一追踪模型。
3. 让知识图谱从静态展示变成可查询的追踪与影响分析底座。
4. 兼容现有 Sprint / Document / Module / FeaturePoint / TestCase / Graph 结构，分阶段渐进改造。
5. 与现有 BF SKILL 流水线对齐，减少 CLI 产物和 TestFlow 平台产物之间的结构割裂。

## 2. 当前结构问题诊断

### 2.1 Document 过重，资产类型表达不足

当前 `documents` 表承担了所有上传文件和 AI 生成文件的记录职责，包括原始需求文档、功能点 Markdown、cases JSON、Excel 汇总等。字段包括 `name`、`file_path`、`file_type`、`content_preview`、`ai_summary`、`keywords`、`module_ids`、`ai_status`、`parse_status`。

问题：

- 所有产物都被当作 Document，缺少资产类型区分。
- 无法表达“这是原始 PRD”“这是 AI 生成的功能点”“这是测试用例 JSON”“这是 OpenAPI 文档”“这是自动化脚本”。
- `module_ids` 用 JSON 存储，只能做弱标签，不能表达强关系和证据来源。
- Document 适合文件管理，不适合承载测试知识网络。

### 2.2 FeaturePoint 过轻，结构化信息丢失

当前 `feature_points` 表主要字段是：

- `name`
- `source_doc_id`
- `sprint_id`
- `module_id`
- `linked_cases`
- `graph_node_id`

但样例项目的 `功能点.md` 中已经包含更丰富的信息：

- 描述
- 操作入口
- 交互元素
- 业务规则
- 优先级
- 来源

当前 Stage 1 在 `pipeline_executor.py` 中提取了这些字段后写入 Markdown，但入库时只保存 `name`，造成重要测试上下文丢失。Stage 2 的 prompt 虽然支持 `description`、`entry`、`elements`、`rules`、`priority`，但实际从数据库读出来的 FeaturePoint 没有这些字段，导致生成测试用例时上下文变薄。

### 2.3 TestCase 缺少 Sprint 和功能点追踪关系

当前 `testcases` 表有：

- `case_no`
- `title`
- `project_id`
- `module_id`
- `module`
- `preconditions`
- `test_data`
- `test_steps`
- `expected_result`
- `actual_result`
- `exec_status`

问题：

- 没有 `sprint_id`，无法区分某条用例属于 sprint0、sprintN 还是 sprint_all。
- 没有 source asset 关系，无法知道用例来自哪个 cases.json、Excel 或 AI 流水线。
- FeaturePoint 与 TestCase 之间只有 `FeaturePoint.linked_cases` 字符串，不可查询、不可统计、不可维护。
- 无法做真实覆盖率，例如“某功能点覆盖了多少正向/异常/边界用例”。

### 2.4 接口文档没有进入知识库结构

样例项目的接口文档有两类：

- `提供的接口文档.md`：人工整理的 API 规划，包含接口状态、优先级、错误码规范、异步执行、回调等需求。
- `Admin API_OpenAPI.json`：OpenAPI 结构化文档，包含大量 path、method、tag、schema。

当前 TestFlow 没有 API Endpoint 一等模型，只能把接口文档当普通 Document。这样无法回答：

- 哪些接口还没有测试用例？
- 哪些 P0 接口未实现或待确认？
- 某个需求变更影响哪些 endpoint？
- 某个测试用例是否覆盖对应 API？
- OpenAPI schema 变化是否影响已有用例？

### 2.5 知识图谱节点和边语义不足

当前 `GraphNode.node_type` 仅注释支持：

- document
- module
- testcase
- feature

当前 `GraphEdge.relation_type` 仅注释支持：

- dependency
- include
- call
- dataflow

问题：

- 节点类型不足以表达需求条目、接口、脚本、执行结果、变更项、UI 页面、UI 元素。
- 关系类型不足以表达 derived_from、covers、tests_api、implements、changes、generated_by 等测试追踪关系。
- `crud_graph.regenerate_graph` 当前只是更新时间和状态，没有真正基于业务数据重建图谱。
- 图谱更像展示页，不是测试影响分析底座。

### 2.6 Sprint 快照和增量关系未结构化

当前 Sprint 有 `is_all` 表示 sprint_all 汇总行，这个方向是正确的。但 Sprint 之间的“增量变更”没有结构化表达。

样例项目中：

- `sprint0` 是初始基线。
- `sprint1`、`sprint2` 是需求变更和新增内容。
- `sprint_all` 是合并后的最新基线。

当前无法结构化表达：

- sprintN 新增了哪些功能点。
- sprintN 修改了哪些已有功能点。
- sprintN 删除或废弃了哪些功能点。
- sprintN 影响哪些已有用例和脚本。

## 3. 总体设计原则

### 3.1 保留现有核心表，渐进扩展

不建议一次性推翻现有模型。保留：

- `projects`
- `sprints`
- `documents`
- `modules`
- `feature_points`
- `testcases`
- `graphs`
- `graph_nodes`
- `graph_edges`

在此基础上新增资产、关系、接口、变更等模型，并扩展现有字段。

### 3.2 文件管理和知识实体分离

Document / KnowledgeAsset 负责“文件与产物”。

FeaturePoint / TestCase / ApiEndpoint / RequirementItem / UIScreen 等负责“知识实体”。

TraceLink 负责“实体之间的可追踪关系”。

### 3.3 图谱从 TraceLink 派生

GraphNode / GraphEdge 不应该成为业务关系的唯一存储，而应该是图谱展示层或缓存层。真实关系优先存在 TraceLink 中，图谱重建时从 TraceLink 和核心实体表派生。

### 3.4 所有 AI 结果必须可追踪来源

AI 提取出的功能点、用例、接口映射关系，都需要记录：

- 来源资产
- 来源片段或文件路径
- 置信度
- 证据文本
- 生成阶段

### 3.5 兼容 SKILL 文件产物结构

当前 BF SKILL 以文件为核心产出：

- `需求文档/{sprint}/需求功能点/{模块}/功能点.md`
- `需求文档/{sprint}/需求功能点/{模块}/cases.json`
- `测试用例/{sprint}/{sprint}_testCase.xlsx`
- `测试用例/{sprint}/{sprint_scripts}/{模块}.spec.ts`
- `测试用例/{sprint}/{sprint_scripts}/selectors/{模块}.selectors.ts`

平台应能导入、解析和追踪这些产物，而不是强迫完全换一套结构。

## 4. 目标数据模型

## 4.1 新增 KnowledgeAsset 表

### 用途

统一记录原始上传文件、AI 生成产物、脚本、接口文档、执行报告等知识资产。

Document 可以继续保留，用于兼容现有文档列表。KnowledgeAsset 作为更明确的资产层，逐步替代 Document 的语义职责。

### 建议模型

新增文件：`testflow-backend/app/models/knowledge_asset.py`

```python
class KnowledgeAsset(Base):
    __tablename__ = "knowledge_assets"

    id = Column(Integer, primary_key=True, autoincrement=True)
    project_id = Column(Integer, ForeignKey("projects.id"), nullable=True)
    sprint_id = Column(Integer, ForeignKey("sprints.id"), nullable=True)
    document_id = Column(Integer, ForeignKey("documents.id"), nullable=True)

    name = Column(String(200), nullable=False)
    asset_type = Column(String(40), nullable=False)
    source_kind = Column(String(40), default="uploaded")
    file_path = Column(String(500), default="")
    file_type = Column(String(40), default="")
    file_size = Column(Integer, default=0)

    module_id = Column(Integer, ForeignKey("modules.id"), nullable=True)
    version = Column(String(40), default="v1.0")
    status = Column(String(30), default="active")
    parse_status = Column(String(30), default="pending")

    content_hash = Column(String(64), default="")
    metadata = Column(JSON, default=dict)
    created_by = Column(Integer, ForeignKey("users.id"), nullable=True)
    created_at = Column(DateTime, server_default=func.now())
    updated_at = Column(DateTime, server_default=func.now(), onupdate=func.now())
```

### asset_type 枚举建议

- `requirement_doc`：原始需求文档。
- `meeting_note`：会议纪要。
- `feature_spec`：功能点 Markdown。
- `test_case_json`：cases.json。
- `test_case_excel`：测试用例 Excel。
- `api_doc_md`：Markdown 接口文档。
- `api_doc_openapi`：OpenAPI JSON。
- `test_script`：自动化测试脚本。
- `selector_map`：选择器文件。
- `execution_report`：执行报告。
- `screenshot`：测试截图。
- `other`：其他。

### source_kind 枚举建议

- `uploaded`：用户上传。
- `ai_generated`：AI 流水线生成。
- `skill_generated`：BF SKILL 生成。
- `imported`：从外部目录导入。
- `manual`：手工创建。

### 需要修改点

- 新增 model。
- 新增 schema：`app/schemas/knowledge_asset.py`。
- 新增 CRUD：`app/crud/crud_knowledge_asset.py`。
- 新增 API：`app/api/v1/knowledge_assets.py`，prefix `/knowledge-assets`。
- 在 `app/models/__init__.py` 注册模型。
- 在 `app/api/v1/router.py` include 新 router。
- 新增 Alembic migration。
- 在 Document 上传、ArtifactManager 保存产物时同步创建 KnowledgeAsset。

## 4.2 扩展 FeaturePoint 表

### 目标

让 FeaturePoint 真正承载功能点结构，而不是只保存名称。

### 建议新增字段

修改文件：`testflow-backend/app/models/feature_point.py`

新增字段：

```python
description = Column(Text, default="")
entry_path = Column(Text, default="")
interaction_elements = Column(Text, default="")
business_rules = Column(Text, default="")
priority = Column(String(10), default="中")
source_type = Column(String(30), default="requirement")
source_asset_id = Column(Integer, ForeignKey("knowledge_assets.id"), nullable=True)
status = Column(String(30), default="active")
version = Column(String(40), default="v1.0")
fingerprint = Column(String(64), default="")
raw_data = Column(JSON, default=dict)
updated_at = Column(DateTime, server_default=func.now(), onupdate=func.now())
```

### 字段说明

- `description`：功能描述。
- `entry_path`：操作入口，例如“系统管理 → 用户管理 → 新增用户”。
- `interaction_elements`：表单字段、按钮、表格列、弹窗等。
- `business_rules`：必填、长度、格式、权限、状态流转等规则。
- `priority`：高/中/低。
- `source_type`：`requirement`、`ui_explore`、`api_doc`、`manual`、`ai_generated`。
- `source_asset_id`：对应 KnowledgeAsset。
- `status`：`active`、`changed`、`deprecated`、`deleted`。
- `fingerprint`：跨 Sprint 匹配同一功能点的稳定指纹。
- `raw_data`：保留 AI 原始结构，便于审计和回放。

### 需要修改点

- 修改 `FeaturePointCreate`、`FeaturePointUpdate`、`FeaturePointOut`。
- 修改 `crud_feature_point.create_feature_point` 和 `update_feature_point`。
- 修改 `api/v1/feature_points.py` 的 `_to_out`。
- 修改前端 `KnowledgeDetail.vue` 功能点列表和创建/编辑弹窗。
- 修改 `pipeline_executor._execute_stage_1`，写入完整字段。
- 修改 `skill_prompts.build_stage2_prompt` 的输入来源，使用完整 FeaturePoint 字段。

## 4.6 新增 TestCaseApiEndpoint 关系表

- 用例步骤或预期中出现接口 path。

## 4.7 新增 TraceLink 通用追踪关系表

### 用途

作为所有知识实体之间的统一关系底座。图谱从 TraceLink 派生。

新增文件：`testflow-backend/app/models/trace_link.py`


- `测试用例/sprintN/*_scripts/*.spec.ts` -> test_script。
- `测试用例/sprintN/*_scripts/selectors/*.selectors.ts` -> selector_map。
- `接口文档/*.md` -> api_doc_md。
- `接口文档/*.json` 且符合 OpenAPI -> api_doc_openapi。

## 14. 风险与注意事项

### 14.1 数据迁移风险

历史数据字段不足，部分关系无法准确回填。处理方式：

- 置信度低的关系设置 `confidence=50~70`。
- evidence 标注“历史数据推断”。
- 前端显示“推断关系”与“确定关系”的差异。

### 14.2 图谱规模风险

OpenAPI 文档可能有数百 endpoint，测试用例也可能很多。图谱一次性展示全部会很乱。

处理方式：

- 图谱接口支持按节点类型和模块过滤。
- 默认展示当前 Sprint 或当前模块的子图。
- 提供节点搜索和上下游展开。

### 14.3 AI 映射误判风险

LLM 将功能点映射到用例、API 时可能误判。

处理方式：

- TraceLink 记录 confidence 和 evidence。
- 低置信度关系需要人工确认。
- 前端提供手动修正关系能力。

### 14.4 与现有页面兼容

新增字段后旧数据可能为空。

处理方式：

- Schema 提供默认值。
- 前端空值显示 `-`。
- API 保持旧字段不删除。

## 16. 推荐结论

推荐将知识库定位从“Sprint 文档快照”调整为“Sprint 测试知识快照”。

核心实体应从：

```text
Sprint -> Document -> FeaturePoint
```

升级为：

```text
Sprint
  -> KnowledgeAsset
  -> FeaturePoint
  -> TestCase
  -> ApiEndpoint
  -> TestScript
  -> ExecutionReport
  -> TraceLink
  -> Graph
```

Document 继续保留，但逐步退回“文件记录/预览”的职责。FeaturePoint、TestCase、ApiEndpoint 和 TraceLink 才是后续测试生成、精准回归、影响分析、知识图谱的核心底座。

这样改完后，类似样例项目这样的真实测试项目就可以被完整表达为：

- 哪个 Sprint 引入了哪些需求。
- 哪些需求形成了哪些功能点。
- 哪些功能点被哪些用例覆盖。
- 哪些用例生成了哪些自动化脚本。
- 哪些接口被哪些需求和用例覆盖。
- 某个变更会影响哪些用例、接口和脚本。
- sprint_all 当前基线是否完整、可测、可回归。

这会比单纯“文档上传 + AI 提取功能点”的知识库更贴近 TestFlow 的产品定位，也更符合 BF SKILL 流水线的真实工作方式。

### 17.1 数据库索引与唯一约束推荐

新增表和扩展字段后，查询会集中在项目、Sprint、模块、实体类型、关系类型、指纹等维度。建议在 Alembic migration 中同步创建索引，避免后续图谱和影响分析变慢。

#### FeaturePoint 索引

推荐索引：

```python
op.create_index("ix_feature_points_project_sprint_module", "feature_points", ["sprint_id", "module_id"])
op.create_index("ix_feature_points_source_asset", "feature_points", ["source_asset_id"])
op.create_index("ix_feature_points_fingerprint", "feature_points", ["fingerprint"])
op.create_index("ix_feature_points_status", "feature_points", ["status"])
```

推荐唯一约束策略：

```text
不建议直接对 name 做唯一约束。
推荐以 sprint_id + module_id + fingerprint 作为同一 Sprint 内功能点去重依据。
```

原因：同名功能点可能出现在不同模块或不同 Sprint；fingerprint 更适合做跨 Sprint 匹配和增量判断。

#### TestCase 索引

推荐索引：

```python
op.create_index("ix_testcases_project_sprint", "testcases", ["project_id", "sprint_id"])
op.create_index("ix_testcases_module", "testcases", ["module_id"])
op.create_index("ix_testcases_case_no", "testcases", ["case_no"])
op.create_index("ix_testcases_case_type", "testcases", ["case_type"])
op.create_index("ix_testcases_automation_status", "testcases", ["automation_status"])
```

推荐唯一约束：

```python
op.create_unique_constraint(
    "uq_testcases_project_sprint_case_no",
    "testcases",
    ["project_id", "sprint_id", "case_no"],
)
```

如果历史数据存在 `sprint_id = null` 的情况，迁移时不要立即加严格唯一约束。推荐先回填 sprint_id，再创建唯一约束；或者第一阶段仅建普通索引，第二阶段清洗数据后补唯一约束。

#### KnowledgeAsset 索引

推荐索引：

```python
op.create_index("ix_knowledge_assets_project_sprint", "knowledge_assets", ["project_id", "sprint_id"])
op.create_index("ix_knowledge_assets_type", "knowledge_assets", ["asset_type"])
op.create_index("ix_knowledge_assets_module", "knowledge_assets", ["module_id"])
op.create_index("ix_knowledge_assets_content_hash", "knowledge_assets", ["content_hash"])
op.create_index("ix_knowledge_assets_document", "knowledge_assets", ["document_id"])
```

推荐唯一约束：

```text
不建议 file_path 全局唯一。
推荐 project_id + sprint_id + asset_type + content_hash 做重复文件检测。
```

原因：同一个文件可能被复制到不同 Sprint；同路径也可能在不同部署环境下不同。

#### ApiEndpoint 索引

推荐索引：

```python
op.create_index("ix_api_endpoints_project_sprint", "api_endpoints", ["project_id", "sprint_id"])
op.create_index("ix_api_endpoints_method_path", "api_endpoints", ["method", "path"])
op.create_index("ix_api_endpoints_module", "api_endpoints", ["module_id"])
op.create_index("ix_api_endpoints_status", "api_endpoints", ["status"])
op.create_index("ix_api_endpoints_tag", "api_endpoints", ["tag"])
op.create_index("ix_api_endpoints_fingerprint", "api_endpoints", ["fingerprint"])
```

推荐唯一约束：

```python
op.create_unique_constraint(
    "uq_api_endpoints_project_sprint_method_path",
    "api_endpoints",
    ["project_id", "sprint_id", "method", "path"],
)
```

如果一个 endpoint 在同一 Sprint 内存在多个版本，应增加 `version` 参与唯一约束，或只保留最新版本并把历史放入 `raw_data` / ChangeItem。

#### TraceLink 索引

TraceLink 是影响分析和图谱重建的核心，必须重点建索引。

推荐索引：

```python
op.create_index("ix_trace_links_project_sprint", "trace_links", ["project_id", "sprint_id"])
op.create_index("ix_trace_links_source", "trace_links", ["source_type", "source_id"])
op.create_index("ix_trace_links_target", "trace_links", ["target_type", "target_id"])
op.create_index("ix_trace_links_relation", "trace_links", ["relation_type"])
op.create_index("ix_trace_links_source_relation", "trace_links", ["source_type", "source_id", "relation_type"])
op.create_index("ix_trace_links_target_relation", "trace_links", ["target_type", "target_id", "relation_type"])
```

推荐唯一约束：

```python
op.create_unique_constraint(
    "uq_trace_links_unique_relation",
    "trace_links",
    ["source_type", "source_id", "target_type", "target_id", "relation_type"],
)
```

如果需要允许多条同类关系但 evidence 不同，可以取消唯一约束，改为应用层 upsert：相同五元组更新 confidence、evidence、metadata。

#### GraphNode / GraphEdge 索引

推荐索引：

```python
op.create_index("ix_graph_nodes_graph_type", "graph_nodes", ["graph_id", "node_type"])
op.create_index("ix_graph_edges_graph_relation", "graph_edges", ["graph_id", "relation_type"])

op.create_index("ix_graph_edges_target", "graph_edges", ["target_node_id"])
```

GraphNode 建议在 `properties` 中保留 `entity_type` 和 `entity_id`，但 PostgreSQL JSON 查询较慢，第一阶段可以不建 JSON 索引。后续如果图谱查询压力大，再考虑 GIN 索引。

### 17.2 权限与多项目隔离策略

根据当前阶段目标，权限和多项目隔离暂时不做强限制。推荐策略如下：

1. API 层仍保留 `get_current_user`，保持登录态要求不变。
2. 新增接口暂不按用户角色过滤项目数据。
3. 所有新增表仍必须保留 `project_id`，为后续权限体系预留。
4. 前端以当前选择项目作为默认筛选条件，但不把它作为安全边界。
5. 后续权限完善时，再统一在 CRUD 层增加 project scope 校验。

也就是说：

```text
当前阶段：登录后可访问平台内数据，前端默认按项目过滤。
后续阶段：再升级为用户/角色/项目成员级权限控制。
```

这样不会阻塞知识库结构重构，也不会给后续权限升级制造障碍。

### 17.3 删除策略推荐

删除策略需要兼顾历史追踪、数据一致性和用户体验。推荐采用“核心实体软删除，关系同步失效，展示默认过滤”的方案。

#### 状态字段约定

建议以下表都使用 `status` 表达生命周期：

- `knowledge_assets.status`：`active`、`deleted`、`archived`。
- `feature_points.status`：`active`、`changed`、`deprecated`、`deleted`。
- `api_endpoints.status`：`existing`、`todo`、`pending_confirm`、`deprecated`、`deleted`、`unknown`。
- `change_items.status`：`pending`、`confirmed`、`applied`、`ignored`。
- `trace_links.metadata.active`：true / false，或新增 `status` 字段。

如果实现成本允许，建议给 TraceLink 也加 `status` 字段：

```python
status = Column(String(30), default="active")
```

#### 删除 Document

推荐行为：

1. `documents` 可以继续物理删除或保留现有删除逻辑。
2. 对应 `knowledge_assets` 不物理删除，改为 `status='deleted'`。
3. 由该 asset 派生的 FeaturePoint / ApiEndpoint 不立即删除，改为保留，并在关系 evidence 中保留来源。
4. 如果用户明确选择“同时删除派生内容”，再将派生 FeaturePoint / ApiEndpoint 置为 `deleted`。

原因：测试知识库需要可追溯历史。删除原始文档不一定代表功能点失效。

#### 删除 Sprint

推荐行为：

- 非 `sprint_all`：允许删除，但需要二次确认。
- 删除 Sprint 时，关联 KnowledgeAsset、FeaturePoint、ChangeItem、TraceLink、Graph 默认软删除或级联删除。
- TestCase 建议保留，但 `sprint_id` 可保留原值；前端默认不展示已删除 Sprint 的用例。

如果当前产品尚未支持软删除 Sprint，可以维持现有物理删除，但新增实体要 cascade 清理，避免孤儿数据。

#### 删除 FeaturePoint

推荐行为：

1. 将 `feature_points.status` 置为 `deleted`。
2. 将关联 `FeaturePointTestCase` 删除或置为 inactive。
3. 将关联 TraceLink 置为 inactive。
4. 不删除 TestCase，因为测试用例可能仍有历史执行价值。

#### 删除 TestCase

推荐行为：

1. 第一阶段沿用现有物理删除。
2. 如果要支持历史审计，后续增加 `status='deleted'` 或 `is_deleted`。
3. 删除时清理 FeaturePointTestCase、TestCaseApiEndpoint、相关 TraceLink。

#### 删除 ApiEndpoint

推荐行为：

- 不物理删除，改为 `status='deleted'` 或 `deprecated`。
- 保留历史覆盖关系，但默认查询过滤。
- 如果 OpenAPI 重新导入时发现接口不存在，推荐标记为 `deprecated`，不要立即 deleted。

### 17.4 OpenAPI 解析细化规则

OpenAPI 文档结构复杂，建议第一阶段实现稳定子集，第二阶段再扩展高级能力。

#### 支持版本

第一阶段支持：

- OpenAPI 3.x：`openapi: 3.0.x / 3.1.x`。
- Swagger 2.0：如果检测到 `swagger: '2.0'`，做基础兼容。

检测规则：

```text
如果存在 openapi 字段 -> OpenAPI 3。
如果存在 swagger 字段 -> Swagger 2。
否则提示“非 OpenAPI 文档”。
```

#### Endpoint 解析粒度

以 method 级别建 ApiEndpoint，而不是 path 级别。

例如：

```text
GET /console/v1/menus -> 1 个 ApiEndpoint
POST /console/v1/menus -> 1 个 ApiEndpoint
PUT /console/v1/menus -> 1 个 ApiEndpoint
```

原因：不同 method 的语义、参数和测试用例完全不同。

#### `$ref` 处理规则

第一阶段推荐做“浅展开 + 保留引用”：

1. 如果 schema 是 `$ref`，解析 `#/components/schemas/Xxx`，取出目标 schema 放入 `resolved_schema`。
2. 原始 `$ref` 仍保留在 `raw_data` 中。
3. 防止循环引用，最大展开深度设为 3。
4. 如果解析失败，保留 `$ref` 字符串，不阻断导入。

伪代码：

```python
def resolve_ref(ref: str, root: dict, depth: int = 0):
    if depth > 3:
        return {"$ref": ref}
    if not ref.startswith("#/components/schemas/"):
        return {"$ref": ref}
    name = ref.split("/")[-1]
    schema = root.get("components", {}).get("schemas", {}).get(name)
    return schema or {"$ref": ref}
```

#### requestBody 解析规则

OpenAPI 3：

优先级：

1. `application/json`
2. `multipart/form-data`
3. `application/x-www-form-urlencoded`
4. 第一个 content-type

保存结构：

```json
{
  "content_type": "application/json",
  "required": true,
  "schema": {},
  "resolved_schema": {}
}
```

Swagger 2：

- 从 `parameters` 中找 `in: body`。
- 其 schema 作为 request_schema。

#### response 解析规则

优先解析：

1. `200`
2. `201`
3. `default`
4. 第一个 2xx
5. 第一个 response

同时保留所有 response 状态码到 `raw_data.responses`。

保存结构：

```json
{
  "primary_status": "200",
  "schema": {},
  "resolved_schema": {},
  "all_status_codes": ["200", "400", "401", "500"]
}
```

#### parameters 解析规则

解析所有：

- path 参数
- query 参数
- header 参数
- cookie 参数

字段：

```json
{
  "name": "id",
  "in": "path",
  "required": true,
  "schema": {"type": "string"},
  "description": "..."
}
```

#### tag 到 Module 的映射

推荐三阶段：

1. 精确匹配：tag 名称等于 Module.name 或 Module.code。
2. 规则匹配：去除编号、空格、特殊字符后匹配。
3. 未匹配：创建或挂到“接口未归类”模块。

不建议第一阶段用 LLM 自动创建大量模块。模块是测试组织维度，过度拆分会影响用例管理。

#### 中文乱码处理

如果 OpenAPI JSON 中 tag 或 summary 出现乱码：

1. 先按 UTF-8 读取。
2. 如果检测到大量 `�`，尝试 `gbk` / `gb18030` 重新读取。
3. 如果仍失败，保留原值，并在导入报告 warnings 中提示。

#### 导入幂等策略

同一个 OpenAPI 重复导入时：

- 以 `project_id + sprint_id + method + path` 查找已有 ApiEndpoint。
- 存在则更新 summary、schema、raw_data、source_asset_id。
- 不存在则新增。
- 本次导入未出现但数据库已有的 endpoint，不立即删除，标记为 `deprecated` 或 `status='unknown'`，并写入导入报告。

### 17.5 图谱查询和性能策略

图谱应支持“全图概览”和“局部分析”两种模式。第一阶段不要默认一次性加载全量节点。

#### 推荐 API

新增或扩展：

```text
GET /api/v1/graphs/{graph_id}
GET /api/v1/graphs/{graph_id}/subgraph
GET /api/v1/graphs/{graph_id}/node/{node_id}/neighbors
GET /api/v1/graphs/{graph_id}/search
```

#### subgraph 查询参数

```text
project_id
sprint_id
module_id
node_types=feature,testcase,api
relation_types=covers,tests_api,implements
center_type=feature
center_id=123
depth=1
limit=300
```

推荐默认：

```text
depth = 1
limit = 300
```

超过 limit 时返回：

```json
{
  "truncated": true,
  "limit": 300,
  "message": "节点数量超过限制，请缩小筛选范围"
}
```

#### 邻居查询规则

`GET /graphs/{graph_id}/node/{node_id}/neighbors?direction=both&depth=1`

- `direction=out`：只查下游。
- `direction=in`：只查上游。
- `direction=both`：上下游都查。
- `depth` 第一阶段最大限制为 2。

#### 前端默认展示策略

1. 图谱列表点击进入时，默认展示模块级概览：module、feature、testcase、api 的聚合视图。
2. 用户点击某个模块后，再展开该模块下的功能点、用例、接口。
3. 用户点击某个节点后，右侧展示一跳上下游。
4. 搜索节点后，展示以该节点为中心的一跳子图。

#### 图谱节点聚合建议

如果某模块下测试用例超过 100 条，默认不展开每条 TestCase，而显示聚合节点：

```text
用户管理测试用例集合（32）
```

点击后再展开明细。

### 17.6 本地项目资料导入任务异步化

本地目录导入可能涉及大量文件、OpenAPI 解析、Excel 解析、脚本扫描，建议不要做同步阻塞接口。

#### 新增 ImportJob 表

新增模型：`app/models/import_job.py`

```python
class ImportJob(Base):
    __tablename__ = "import_jobs"

    id = Column(Integer, primary_key=True, autoincrement=True)
    project_id = Column(Integer, ForeignKey("projects.id"), nullable=False)
    root_path = Column(String(500), nullable=False)
    job_type = Column(String(40), default="local_project")
    status = Column(String(30), default="pending")
    dry_run = Column(Boolean, default=True)

    total_files = Column(Integer, default=0)
    processed_files = Column(Integer, default=0)
    success_count = Column(Integer, default=0)
    failed_count = Column(Integer, default=0)

    result_summary = Column(JSON, default=dict)
    warnings = Column(JSON, default=list)
    errors = Column(JSON, default=list)
    created_at = Column(DateTime, server_default=func.now())
    started_at = Column(DateTime, nullable=True)
    completed_at = Column(DateTime, nullable=True)
```

#### 状态枚举

- `pending`
- `running`
- `succeeded`
- `failed`
- `partial_success`
- `cancelled`

#### API 设计

```text
POST /api/v1/import-jobs/local-project
GET /api/v1/import-jobs
GET /api/v1/import-jobs/{job_id}
POST /api/v1/import-jobs/{job_id}/cancel
```

创建任务请求：

```json
{
  "project_id": 1,
  "root_path": "D:/BF-VAgent",
  "dry_run": true,
  "options": {
    "import_requirements": true,
    "import_testcases": true,
    "import_api_docs": true,
    "import_scripts": true
  }
}
```

#### dry_run 推荐流程

1. 扫描目录。
2. 分类文件。
3. 检查可解析性。
4. 返回预计导入资产数量、Sprint 数量、风险和 warning。
5. 不写入业务表。

#### 正式导入推荐流程

1. 创建 ImportJob，状态 pending。
2. 后台任务更新为 running。
3. 逐文件处理，每处理一个文件更新进度。
4. 单文件失败不终止整个任务，记录 errors。
5. 结束时：
   - 全部成功：succeeded。
   - 部分失败：partial_success。
   - 致命错误：failed。

#### 导入幂等策略

- KnowledgeAsset 用 `content_hash` 去重。
- FeaturePoint 用 `sprint_id + module_id + fingerprint` upsert。
- TestCase 用 `project_id + sprint_id + case_no` upsert。
- ApiEndpoint 用 `project_id + sprint_id + method + path` upsert。
- TraceLink 用五元组 upsert。

### 17.7 测试脚本与用例精确映射规则

Playwright 脚本需要和 TestCase 建立稳定关系。推荐以 case_no 为主，文件路径和模块为辅。

#### spec 文件识别

路径规则：

```text
测试用例/{sprint}/{scripts_dir}/{模块名}.spec.ts
测试用例/{sprint}/{scripts_dir}/selectors/{模块名}.selectors.ts
```

导入时创建：

- spec.ts -> KnowledgeAsset `test_script`。
- selectors.ts -> KnowledgeAsset `selector_map`。

#### test() 标题解析规则

优先匹配：

```typescript
test('VAgent_TC_YHGL_001 - 进入用户管理页面并查看用户列表', async ({ page }) => {})
```

正则：

```text
([A-Za-z0-9]+_TC_[A-Za-z0-9]+_\d{3,})
```

匹配到 case_no 后：

1. 查找同 project_id + sprint_id + case_no 的 TestCase。
2. 建立 `testcase -> script` TraceLink，relation_type=`generated_by`。
3. 更新 TestCase：
   - `automation_status='generated'`
   - `automation_path=spec_path`

#### 一个 spec 多条用例

一个 spec 文件通常包含多条 test()。处理方式：

- KnowledgeAsset 只创建一个 script asset。
- 每个 TestCase 都通过 TraceLink 指向同一个 script asset。
- TraceLink metadata 中记录 test block 名称和行号。

示例：

```json
{
  "test_title": "VAgent_TC_YHGL_001 - 进入用户管理页面并查看用户列表",
  "line": 42
}
```

#### 无 case_no 的 test()

如果 test 标题没有 case_no：

1. 尝试用标题与 TestCase.title 精确匹配。
2. 再尝试同模块标题相似度匹配。
3. 匹配成功但不确定时，TraceLink confidence 设为 60。
4. 匹配失败时，创建 warning，不自动关联。

#### selector 关联规则

- spec asset -> selector asset，relation_type=`depends_on`。
- 如果 spec 中 import 路径明确指向 selector 文件，confidence=100。
- 如果按模块名推断，confidence=70。

### 17.8 覆盖率计算规则

建议统一平台内覆盖率口径，避免不同页面统计不一致。

#### 功能点用例覆盖率

```text
已覆盖功能点数 / active 功能点总数
```

已覆盖定义：

- 存在 active 的 FeaturePointTestCase；或
- 存在 active TraceLink：feature -> testcase，relation_type=covers。

#### API 用例覆盖率

```text
已覆盖 API 数 / active API 总数
```

已覆盖定义：

- 存在 TestCaseApiEndpoint；或
- 存在 active TraceLink：testcase -> api，relation_type=tests_api。

#### 自动化覆盖率

```text
已生成自动化脚本的测试用例数 / 测试用例总数
```

已生成定义：

- `testcases.automation_status in ('generated', 'validated', 'executed', 'healed')`；或
- 存在 testcase -> script 关系。

#### 回归影响覆盖率

用于 Sprint 增量：

```text
已处理影响项 / 总影响项
```

已处理定义：

- ChangeItem 已关联至少一条新增或更新用例；或
- ChangeItem.status in ('applied', 'ignored')。

### 17.10 实施优先级补充建议

如果按“最快提升平台价值”排序，推荐：

1. FeaturePoint 保真入库。
2. TestCase 增加 sprint_id。
3. FeaturePointTestCase 覆盖关系。
4. Sprint knowledge overview 聚合接口。
5. KnowledgeAsset 资产层。
6. OpenAPI 解析和 ApiEndpoint。
7. TraceLink。
8. 图谱从 TraceLink 重建。
9. ImportJob 异步导入。
10. ChangeItem 增量影响分析。

其中 1 到 4 是第一批必须优先做的，因为它们直接决定 AI 生成的内容能否沉淀到平台。5 到 8 是知识图谱真正变得有价值的关键。9 和 10 更偏后续平台成熟后的增强能力。
