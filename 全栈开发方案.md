# TestFlow 全栈开发方案 — 基于功能描述的前后端设计

## Context

原型文件夹 `Prototype/TestFlow/功能描述.md` 定义了完整的交互功能规格。当前后端数据模型为 `KnowledgeBase → Folder → Document` 三层结构，但前端页面已按 `Project → Sprint → Module → Document + FeaturePoint` 新模型重构（使用 Mock 数据）。核心矛盾是**前后端数据模型不匹配**，需要重构后端模型并提供真实 API。

**当前后端已有的模块**（无需改动）：Auth、User、Role、Project、Activity、Dashboard
**需要重构的模块**：Knowledge（→ Sprint + Module + Document）
**需要新建的模块**：AI Config、Knowledge Graph、AI Pipeline、Global Search
**需要补充字段的模块**：TestCase（+3字段）、Report（+4字段）

## Phase 1: Sprint/Module/Document 模型重构

**目标**：用 Sprint + Module 替换 KnowledgeBase + Folder，Document 外键从 folder_id 改为 module_id。

### 新模型定义

**Sprint** (`app/models/sprint.py` — 新建)
```
sprints
├── id: Integer, PK
├── name: String(100), NOT NULL
├── description: Text
├── project_id: Integer, FK → projects.id
├── status: String(20), 默认"待启动"  # 基线/已完成/进行中/待启动/最新汇总
├── is_all: Boolean, 默认 False       # sprint_all 汇总行
├── created_at: DateTime
└── updated_at: DateTime
```

**Module** (`app/models/module.py` — 新建)
```
modules
├── id: Integer, PK
├── name: String(100), NOT NULL
├── sprint_id: Integer, FK → sprints.id
├── ai_status: String(20), 默认"待分析" # 已分析/分析中/待分析
├── created_at: DateTime
└── updated_at: DateTime
```

**Document** (`app/models/document.py` — 新建，替代 knowledge.py 中的 Document)
```
documents
├── id: Integer, PK
├── name: String(200), NOT NULL
├── file_path: String(500)
├── file_type: String(20)              # PDF/Word/Markdown/Excel
├── file_size: Integer
├── module_id: Integer, FK → modules.id  # ← 原 folder_id 改为 module_id
├── uploader_id: Integer, FK → users.id
├── version: String(20), 默认 "v1.0"    # 新增
├── content_preview: Text               # 新增
├── ai_summary: Text                    # 新增
├── keywords: JSON                      # 新增
└── created_at: DateTime
```

### 现有模型字段补充

**TestCase** (`app/models/testcase.py` — 修改)
- 新增 `preconditions: Text` — 前置条件
- 新增 `test_steps: Text` — 测试步骤
- 新增 `expected_result: Text` — 预期结果

**Report** (`app/models/report.py` — 修改)
- 新增 `report_type: String(20)` — 回归/冒烟/迭代/全量
- 新增 `test_scope: Text` — 测试范围描述
- 新增 `approved_by: Integer, FK → users.id` — 审批人
- 新增 `approved_at: DateTime` — 审批时间

### 迁移策略：Clean Break

项目早期无生产数据，采用**一次性迁移**：
- 删除旧表 `knowledge_bases`、`folders`
- 创建新表 `sprints`、`modules`
- 修改 `documents` 表：删除 `folder_id` 列，新增 `module_id` 列和版本字段
- 修改 `testcases` 表：新增 3 个字段
- 修改 `reports` 表：新增 4 个字段

## Phase 4: AI 配置模块

### 新模型（全部在 `app/models/ai_config.py`）

**AIProvider** — AI 服务商
```
ai_providers
├── id, provider_type(OpenAI/Anthropic/DeepSeek/Custom), name, api_key,
├── model, endpoint_url, temperature(Float), max_tokens(Integer),
├── status(正常/限流/错误), last_call_at, created_at
```

**ModelStrategy** — 模型分配策略
```
model_strategies
├── id, task_type(unique, 5种), provider_id(FK), model_name, temperature(Float)
```

**AIGlobalConfig** — 全局参数
```
ai_global_configs
├── id, key(unique), value
```

**AICallLog** — 调用日志
```
ai_call_logs
├── id, task_type, model, provider_id(FK), input_tokens, output_tokens,
├── duration_ms, status(成功/失败/超时), created_at
```

### API 端点（`/api/v1/ai`）
| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/providers` | 服务商列表 |
| POST | `/providers` | 新建服务商 |
| PUT | `/providers/{id}` | 编辑服务商 |
| DELETE | `/providers/{id}` | 删除服务商 |
| POST | `/providers/{id}/test` | 测试连接 |
| GET | `/strategies` | 策略列表 |
| PUT | `/strategies` | 批量更新策略 |
| GET | `/config` | 全局参数 |
| PUT | `/config` | 更新全局参数 |
| GET | `/stats` | 统计 |
| GET | `/call-logs` | 调用日志 |

### 种子数据
- AIGlobalConfig: max_tokens=4096, timeout=60, retries=3, concurrency=5, log_level=INFO
- ModelStrategy: 5 种任务类型各一行

## Phase 6: AI 流水线 / 工作台

### 新模型（`app/models/pipeline.py`）

**PipelineExecution**
```
pipeline_executions
├── id, project_id(FK), sprint_id(FK), mode(full/incremental),
├── status(running/paused/completed/failed/waiting),
├── started_at, completed_at, total_duration_ms
```

**PipelineStage**
```
pipeline_stages
├── id, execution_id(FK), stage_no(1-4), stage_name,
├── status(completed/running/waiting/failed),
├── model, input_tokens, output_tokens, duration_ms,

```

### API 端点（`/api/v1/pipeline`）

## 验证方式

每个 Phase 完成后：
1. **后端**：运行 `alembic upgrade head`，启动 FastAPI，访问 `/docs` 验证新接口
2. **前端**：部署到服务器，替换对应页面的 mock 数据为真实 API 调用
3. **端到端**：在浏览器中操作页面，验证 CRUD 流程完整

关键验证点：
- Phase 2A：知识库 Sprint/模块/文档三层导航全部连通
- Phase 2B：测试用例展开行显示完整字段；报告可审批
- Phase 4：AI 配置页服务商可增删改查，测试连接按钮有响应
- Phase 6：AI 工作台可启动/暂停执行，阶段状态实时更新
